## 步骤

### Step 1：修复 WXT 构建冲突（P0，~10 min 人工）

**选择方案 A（推荐）**：将测试文件移出 entrypoints 目录。

操作：

- 新建 `__tests__/entrypoints/` 目录
- 移动 `entrypoints/background.test.ts` → `__tests__/entrypoints/background.test.ts`
- 更新 `vitest.config.ts` 的 `include` 确保 vitest 仍能找到该测试
- 验证：`pnpm build` 成功，产出 `.output/chrome-mv3/`

备选方案 B（不推荐但可作为备选）：

- 将 `entrypoints/background.test.ts` 重命名为 `background.spec.ts`
- 在 `vitest.config.ts` 里显式 include `**/*.spec.ts`
- 风险：WXT 仍有可能会扫到 `.spec.ts`，不如移出目录干净。

**验证标准**：

```bash
pnpm build   # 无报错
pnpm zip     # 成功产出 zip
ls .output/chrome-mv3/  # 存在 manifest.json
```

### Step 2：补齐 Side Panel 使用文档（P1，~20 min 人工）

操作：

- 更新 `README.md` "使用" 章节：描述新的 Side Panel 视图（Batch 视图、档位切换、打字手势确认 authorized、填充结果三态表、History 标签页）
- 补充 `docs/batch-usage-guide.md`（新建）：
  - 档位说明：off / dry-run / authorized 的适用场景
  - tab 漂移阻断：橙标出现时切回授权标签页
  - 隔离仓处置流程：needs-human-verification 条目的正确处理方式
  - dry-run 与 authorized 真发的操作路径

**验证标准**：

- README 中单条流程描述替换为 batch 流程
- 新增文档完整覆盖 3 个档位的使用方式

### Step 3：端到端真人验证（P2，~30 min 人工）

操作：

- 在 Chrome 加载 `.output/chrome-mv3/`
- 打开 `https://dx-999-adm.ympxbys.xyz/` 后台发帖页
- 打开 Side Panel → 进入 Batch 视图
- 走以下链路并记录结果：
  1. 输入 1-2 个主题 → 生成草稿 → 确认进入 `awaiting-approval`
  2. 切换到 `dry-run` 档位 → 批准 → 查看 DryRunReport 填充结果
  3. 切换到 `authorized` → 输入 `publish` → 批准
  4. 观察填充结果：Check FillStatusTable 三态
  5. 在后台核对发布结果是否与 Side Panel 一致
  6. 进入 History Tab → 确认轨迹记录出现
  7. 刷新 Side Panel → 确认 batch 状态持久化恢复
  8. 关闭标签页 → 重新打开 → 确认崩溃恢复自动处理 tombstone

**验证标准**：

- 填充结果三态正确（filled / degraded 含义清晰）
- authorized 模式下真实发布成功
- History Tab 轨迹记录完整且可校验
- 刷新 / 重启后状态正确恢复

### Step 4：dry-run 策略文档化（P3，~10 min 人工）

操作：

- 在 `docs/batch-usage-guide.md` 或新建 `docs/dry-run-strategy.md` 中明确：
  - `saveDryRunReport` 触发时机：approveBatch 完成 dry-run 批准后
  - 存储位置：`local:dryRunReport`（batchId + ts + items）
  - 持久化时机：best-effort，失败不阻断流程
  - 清理时机：新批次批准时自动覆盖 / 手动清空后清除
  - Side Panel `DryRunReport` 组件读取逻辑对齐

**验证标准**：

- 文档中 dry-run 流程与代码逻辑一致
- Side Panel 展示的报告数据与实际存储匹配

## 风险

- **Step 1 的 include 扩散风险**：当前 vitest.config 已扫描 `**/*.test.ts` 和 `**/*.spec.ts`，如果还有其他测试文件藏在 `entrypoints/` 下（如 `content.test.ts`、`sidepanel/xxx.test.tsx`），需要一并移出。**注**：当前仅有 `background.test.ts` 一个冲突文件。
- **README Batch 流程指令更新不完整风险**：如果运营人员按 README 没看到 Batch 入口，会困惑。需确保 Side Panel 的 batch 入口按钮文案清晰。
- **端到端 dry-run / authorized 切换需人工确认**：这是设计行为（安全），但若 Side Panel 在切换档位后状态未同步，会导致流程卡住。
