# 学习仪器:把「审改 → 真发」流程变成自动复利的反馈闭环

> 承接 `2026-06-05-release-readiness-gap-list-requirements.md`。
> 那份盘缺口,结论是「引擎过度建设、唯一真闸是操作者那次首飞」。
> **这份是同一天的下一步选择**:不再加生成功能,而是把这台「不会学」的引擎,
> 改成趴在你反正要走的流程上、零额外操作、自动攒 GATE 数据并越用越准的**学习仪器**(L2 闭环)。

## Problem Frame

谁受影响:**单操作者(自用,每天靠它发本站帖)**。

现在的两个痛:
1. **GATE 卡死两天** —— 合格率 + 计时(go/no-go 的核心判据)只能靠操作者**手填一张纸表**(`stage0-baseline-worksheet.md`),摩擦太大、一直没填。
2. **引擎不会学** —— 每次生成是**孤立**的:操作者改了什么、在后台手补了哪个分类/标签、踩了哪次降级,全部蒸发,下一篇从零开始。

变成什么:从「一台不会学的引擎 + 一张没人填的纸表」→「**一台趴在审改→填充→真发流程上、极少额外操作就自动攒数据、且每篇真发都让下一篇更准的仪器**」。⚠ **前提(document-review 点破)**:仪器只在操作者**真的在跑这条流程**时才有数据——若卡点是「迟迟没跑过一次真题材」(已 2 天),先建仪器并不能解冻 GATE,只是把同一个摩擦挪到「去跑 5–10 条」后面。

为什么这是当下最高杠杆:它**不加任何生成功能**(引擎已过度建设),却同时
- **解冻 GATE**:在 `off`/`dry-run` 模式就能自动攒合格率/计时——**不需要那个可怕的不可逆 authorized 按钮**;
- **把两个卡住的缺口当副产品解掉**:G1(基线数据)和 G4(分类/标签词表)都从「操作者额外手工劳动」变成「正常用就顺手撈到」。

## User Flow(捕获点标注)

```
 ┌─────────────────────────────────────────────────────────────┐
 │  生成 ─→ 审改 ─→ 填充 ─→ [off / dry-run]  或  [authorized 真发]│
 │   │       │       │            │                              │
 │   │       │       │            └─→ 真发结果/轨迹(已有)        │
 │   │       │       └─ [捕获] 降级的分类/标签 + 你随后手选的值    │ ← G4 词表
 │   │       └─ [捕获] 编辑增量(原稿↔终稿)→ 直发/小改/不合格     │ ← G1 合格率
 │   │          [捕获] 审改耗时 + (手输)找事实耗时               │ ← G1 计时
 │   └─ [快照] AI 原稿(留作差异基准)                            │
 └─────────────────────────────┬───────────────────────────────┘
                               ▼
                   📊 看板:合格率 · 净耗时 vs 手工 · 降级率趋势
                               │
                               ▼
            💡 一键回灌建议(你拍板,绝不静默改):
               · 这条「直发」好稿 → 存进 fewshot 口吻范例?
               · 「成人動畫→2」这条映射 → 存进分类/标签词表?
                               │
                               └──→ 下一篇生成更准(越用越准)
```

> 关键:**整条捕获链在 `off`/`dry-run` 就全部生效**——跑 5–10 条真题材即可喂饱 GATE,
> authorized 真发(不可逆、只有操作者能按)可另行择机,不卡这台仪器的主价值。

## Requirements

**捕获(L1 基础 · 被动 · 零额外操作)**
- R1. **快照 AI 原稿**:生成时留存 AI 原始草稿,与操作者最终稿(`draftOverrides`)分开存,作编辑增量的差异基准。
- R2. **编辑增量 → 质量自判**:按「原稿 ↔ 终稿」差异自动判 **直接发(零改)/ 需小改(局部改)/ 不合格(大改或弃发)**;操作者可一键改判(自动只是建议)。
- R3. **计时**:自动记「生成耗时」「草稿停留=审改耗时」;**「找事实耗时」是唯一手输**(发生在扩展外,无法自动捕获)——给一个最轻的入口(一个分钟数 / 一键起停),GATE 判据必需。
- R4. **降级词捕获**:每次 fill 把 `degraded` 的分类/标签原始自由文字(已在 `FieldFillResult.note`,如「无匹配选项:成人動畫」)连同操作者随后在后台真选的 value 落盘;`off`/`dry-run` 也记。
- R5. **不依赖真发即生效**:R1–R4 全程在 `off`/`dry-run` 工作——5–10 条真题材在 `off` 模式跑完就够做 GATE 判定,不需 authorized。

**仪表盘(L1 · 判读)**
- R6. **看板**:展示 合格率(直发+小改 %)、净耗时分解(找事实 / 审改,对比手工基线)、降级率趋势;字段直接对上 `stage0-baseline-worksheet.md` 的 GATE 判据,**替代那张手填纸表**。
- R7. **手工基线录入**:一次性录几条「纯手写同题」的计时,作为「vs 手工」的对照线(否则「省不省时」无从判)。

**回灌(L2 · 人在环 · 一键复利)**
- R8. **词表回灌(= G4,高置信先做)**:把捕获的「自由文字 → 操作者真选 value」映射,一键 promote 进分类/标签词表;之后生成端约束或填充端归一化用它,**降级率自降**。这是唯一有硬实测证据、且每篇都掉质的缺口。
- R9. **范例回灌(口吻 fewshot)**:把判为「直接发」的优质终稿,一键 promote 进 `Settings.fewShotExamples`;**越用口吻越稳**(软收益,次于 R8)。
- R10. **每步人拍板**:所有回灌都是「建议 → 你确认」,**绝不静默改 prompt/词表**——这是 L2 与 L3 的硬边界。

## Success Criteria

- **GATE 解冻**:操作者在 `off` 模式跑 5–10 真题材后,看板**自动**给出合格率 + 净耗时,无需手填纸表 → go/no-go 有据可判。
- **复利可见**:连续用 N 篇后有趋势数据证明「越用越准」——降级率下降(R8 生效)、口吻一致性上升(R9 生效)。
- **额外操作极少(已据 review 下修「零额外操作」措辞)**:每篇一个「找事实耗时」手输 + 偶尔一键 promote(R8 还须手补后台所选 value)+ R7 一次性手工基线;数据捕获本身不需专门动作,**但前提是操作者真的在跑这条流程**。
- **不掉质、不失控**:回灌全程人拍板,**没有一次**静默改写 prompt/词表。

## Scope Boundaries(本轮明确不做)

- **不做 L3 自动学习**:在未验证的引擎上自动回灌 = 放大坏模式风险。
- **不加任何新生成能力 / 可靠性 / 可观测性**:引擎已过度建设,封存。
- **不替操作者做 authorized 真发**:仍只有操作者能按那个不可逆按钮;但本仪器主价值在 `off`/`dry-run`,**刻意不被这个 blocker 卡住**。
- **不做跨站 / 多操作者 / 云端聚合**:单操作者本地 `chrome.storage.local`。
- **不为「找事实耗时」造复杂计时器**:它发生在扩展外,一个粗略手输即可。

## Key Decisions

- **趴在「反正要走的流程」上** → G1(基线)与 G4(词表)当副产品解掉,零额外操作。这是本方向比「再加功能」高杠杆的根因。
- **主价值落在 `off`/`dry-run`** → 解冻 GATE **不需要**那个可怕的不可逆按钮;G2 首飞解耦、可另行择机。
- **选 L2 而非 L3** → 复利但人在环;未验证引擎上不容静默漂移。
- **词表回灌(R8)先于范例回灌(R9)** → G4 有硬实测证据(每篇都降级),fewshot 是软收益。
- **用编辑增量当质量信号** → ⚠ **核码后下调可信度**:① AI 原稿当前被覆盖、须新增留存(见 Dependencies ①);② 操作者若在**后台 Quill** 改正文(扩展看不到回读),edit-delta=0 会把它误判「直发」;③ 一篇**流畅的幻觉**操作者没抓到→零改→也被判「直发」——edit-delta 量的是「改了多少」,不是「对不对」。故质量判定**须以人工一键判定为主、edit-delta 仅作默认**,不能单凭它过 GATE。

## Dependencies / Assumptions

- ⚠ **数据结构只部分支撑(document-review 核码修正,2026-06-05)**:**仅** `FieldFillResult.note`(降级自由文字)与 `SafetyMode` 的 `off`/`dry-run` 现成可用。其余三处均为**净新增工程,非复用**:
  - ① R1 的 AI 原稿**未单独留存**——`patchBatchDrafts` 把 override **原地覆盖** `item.draft`,原稿被销毁;算编辑增量须**新增**持久化字段(如 `BatchItem.aiDraft`)。
  - ② R3 的计时字段**不存在**——只有 `createdAt`(生成时刻),无 shownAt/approvedAt;且 `appendTrajectory` **仅 authorized 落档**,`off`/`dry-run` 当前**无处可落** edit-delta/计时,须**新增** off 模式度量存储。
  - ③ R8 要的「操作者随后在后台真选的 value」**不可观测**——content 脚本填完**不回读表单**(无 MutationObserver),`dry-run` 更不调 `executePublish`;操作者的后台手选扩展**看不到**。
- 假设操作者愿为每篇花一两秒手输「找事实耗时」(GATE 判据必需,否则净耗时结论无效)。
- 词表回灌走「**按需增量学**」而非一次抓全表(后台标签已 3912 个,规模大;brainstorm 禁抓取)。⚠ 但因后台手选不可观测(见上 ③),**「正常用就喂进来」不成立**——操作者须在 promote 时**手动补**所选 value。
