# 距离「每天能用」的完整优先级缺口清单

> 配套上一轮收敛文档 `2026-06-05-content-quality-and-first-flight-requirements.md`。
> 那份定方向(源接地·先基线再建),**这份把「距离可发布/每天能用」的所有缺口一次性盘成带优先级 + 工作量 + 归属的清单**,供操作者挑做。

## Problem Frame

51publisher 工程上是一台造得过度精致的发动机:286 单测 + 23 e2e 全绿、批量状态机 / 轨迹哈希链 / 崩溃恢复 / TOCTOU 修复 / dry-run / 源接地防幻觉(真模型验证「零连结编造」通过)全部落地。

**但它从没真正点过火。** authorized 真实发布一次没跑通,合格率基线从没在本站真题材上测过。这台引擎 **100% 造好、~0% 真实验证**。

因此「更接近可以发布」缺的不是功能。把缺口诚实盘开,只有四类:**(P0) 只有操作者能做的真实验证**——这是真正的发布闸;**(P1) 我能做、且能让那次首飞真的成功的首飞前修复**(两处有硬证据的缺口);**(P2) 运维/安全杂务**;**(P3) 验证证明需要后再做的打磨**(不预先建)。

核心张力不是「加什么」,而是 **P1 该在首飞前修、还是让一次「脏首飞」先暴露真正会坏的地方再修**——见 Key Decisions 与 Resolve Before Planning。

## Requirements

**P0 — 真实验证(唯一真正的发布闸,只有操作者能做)**
- G1.(前提基线,闸门依据)用现版 prompt 在 5–10 个本站真实题材上生成草稿,**分开记**「直接发 / 需小改(时间界定)/ 不合格」,并对每条计时:事实搜集耗时(记事实从哪来)+ 审改耗时;同批记一个手工撰写基线对比。产出决定「找事实是否才是真瓶颈」的数据。
- G2.(首飞,真发布闸)走通**至少一条 authorized 真实发布**完整链路(生成 → 填充 → `POST /admin/webarticle/save` → 后台核对 → `verifyTrajectory()` ✓),全程记录摩擦点(选择器漂移、Quill 真实行为、layui 提交、cover_url 是否必填)。

**P1 — 首飞前修复(我能做,让 G1/G2 真的成功;均有硬代码证据)**
- G3.(后台漂移回填)`lib/field-mapping.ts:6–16` 当前**无 `cover_url` 字段**,而 2026-06-04 后实证后台新增了 `cover_url`(hidden)、tags 涨到 3912、品牌名改「海角社区」+ 加载 hls.js。须重抓一份脱敏 fixture 快照 → 看 contract 测试红 → 据实回填 field-mapping / recipe。**前提:先确认 cover_url 是否为 save 必填**(由 G2 脏首飞或操作者核实)。
- G4.(分类/标签真实词表映射)`lib/fillers.ts:31–42 / 59–82`:分类按 option value/文本匹配、标签按 label 文本匹配。真模型验证里模型吐自由文字「成人動畫」「校園/日常」→ 后台无此选项 → 返回 `degrade`,**帖子照发但缺分类/标签**。须:① 操作者提供后台真实分类(type 2 漫畫/4 動漫…)与标签词表;② 加一层「模型自由文字 → 后台 value」的归一化/映射(或在 prompt 端约束模型只从清单选)。这是唯一有实测证据、且降低每一篇真帖质量的缺口。

**P2 — 运维 / 安全卫生(非能力交付,但发布前该清)**
- G5. 轮换验证期间曝光的 API key,并复核 key 权限最小化(该端点仅 2 个模型对此 key 可用,符合最小权限)。
- G6. 删后台遗留测试帖 id 110/111/112(标题带 `TEST_勿用`)。
- G7. 决定 `scripts/probe-grounding.mjs`(无机密的验证脚手架)入库还是 .gitignore,清掉 git status 噪音。

**P3 — 打磨(验证证明需要后再做,本轮不预先建)**
- G8. Settings 加 few-shot 范例编辑 UI(默认范例已生效,仅在 G1 证明需调口吻时才补编辑器)。
- G9. 迭代时保存上轮草稿做并排 diff / 版本暂存(R8 已有 bypassReentry 重跑通道;仅在首飞证明「肉眼比对两轮输出」是瓶颈时才建)。
- G10. 事实块输入体验打磨 + 净耗时计时辅助(仅在 G1 证明「贴事实/找事实」是日常摩擦时才投入)。

## Success Criteria

- **发布就绪 = G2 跑通**:至少一条 authorized 真实发布后台保存成功、扩展状态与后台一致、轨迹可校验。
- **每天能用 = G1 数据达标**:「直接发 + 需小改」≥ 80%(待操作者拍板),且真实题材单帖净耗时(含找事实+贴+审改)≤ 手工撰写同题——否则即便能发也不算「替代手工」。
- **真帖不掉质 = G4 闭合**:首飞后抽查,分类/标签不再因自由文字被 `degrade` 丢弃。

## Scope Boundaries(本轮明确不做)

- 不加任何新可靠性 / 可观测性(已过度建设,封存)。
- 不做多站点 / 配置化 / 对外分发 / 直接 API 发布器(确认单操作者自用)。
- 不做源数据自动抓取 / 链接可达性网络探测(MV3/CORS 受限;事实由操作者贴)。
- P3 项**不预先建**——等 G1/G2 真实数据证明需要。

## Key Decisions

- **「可发布」的真闸是 G2,不是任何功能**:四次评审已判定工程过度建设,本清单刻意把 P0 摆在功能之前。
- **P1 是「让首飞成功」而非「再加功能」**:G3/G4 都是已存在代码里的真缺口(field-mapping 缺字段、filler 对自由文字降级),不是新能力。
- **倾向「脏首飞优先」**:G2 可用现版代码先打通管路,让真实后台告诉我们 cover_url 到底必不必填、分类 degrade 能不能接受;再据此决定 G3/G4 做多深。避免对「cover_url 必填」这类**假设**预先投资。这是本轮唯一需要操作者拍板的排序问题。

## Dependencies / Assumptions

- G3 假设后台 `POST /admin/webarticle/save` 字段集自 2026-06-04 实证以来除已知三处漂移外未再变(G2/G3 核实)。
- G4 依赖操作者提供后台真实分类/标签词表(brainstorm 禁抓取,须人工给)。
- G1 假设操作者愿意每条帖手动提供核心事实以换取零幻觉。

## G2 脏首飞执行单(照着点 · 用现版代码 · 目标:打通一条真发并暴露真相)

> 前提:扩展已 `pnpm build` 加载;Settings 里 endpoint/key/模型已配、能「↻ 拉取模型列表」。
> 后台:`https://dx-999-adm.ympxbys.xyz/` 发帖页已登录(登录你自己来,我绝不替输密码)。
> **建议先发一条 status=隐藏 的真帖**——真走 `POST /save`(真验证管路),但读者看不到,出问题可删。确认无误再发显示帖。

**Step 1 · 生成(side panel ≣ 批量)**
1. 选题框贴 **1 条**本站真题材(用 `stage2-validation-pack` 的分隔符语法,连结/事实填真值)。
2. 「开始批量(生成+填充)」→ 展开草稿,核对:作品名/集数/连结是否=你输入(逐字)、无【待补】残留、口吻对。

**Step 2 · 填充 + 盯三件事(关键观测点)**
3. 把后台「添加」表单打开(side panel 作用于当前标签页)→ 触发填充 → 看「填充结果」三态表:
   - 🔴 **分类 / 标签是不是 `degrade`(无匹配选项/标签)?** → 这就是 G4。记下后台真实分类项(漫畫/動漫…)和标签清单,贴回来。
   - 🟡 **封面 / cover_url**:后台保存时会不会因为没封面/没 cover_url 报错或存成空?→ 这决定 G3 要不要做、cover_url 填什么。
   - 其余字段(标题/副标题/描述/media_id/status/published_at)是否都绿。

**Step 3 · 真发(authorized · 不可逆 · 你亲手)**
4. Settings 档位切 `authorized`;**先建议把该帖 status 设「隐藏」**。
5. 批准 → 按提示打 `publish` 手势确认 → 观察:
   - `POST /admin/webarticle/save` 是否 `{code:0,msg:操作成功}`(F12 Network 看)。
   - 后台列表是否真出现这条、内容/分类/标签/封面与预期一致。
   - 「历史」标签轨迹是否记录、`verifyTrajectory()` ✓。

**Step 4 · 贴回来给我(决定 G3/G4 怎么修)**
6. 回报四件事:① 分类/标签有没有 degrade + 后台真实词表;② cover_url 必不必填;③ save 是否成功 + 后台实际呈现;④ 全程最大摩擦点。
7. 清理:若发的是测试性隐藏帖,连同遗留的 id 110/111/112 一起删(搜 `TEST_勿用`)。

> 跑通这一条 = G2 闭合 = 发布闸过。随后并行做 G1 基线(`stage0-baseline-worksheet.md`)攒「每天能用」的合格率/省时数据。
