# 内容质量 & 首飞验证(收敛打磨,接近"每天能用")

## Problem Frame

51publisher 已是一台**打磨精致的发动机**:v1 + plan-004 + plan-005 全部落地,
217 测试全绿,可 `pnpm build` 出扩展,批量状态机 / 轨迹哈希链 / 崩溃恢复 / TOCTOU 修复
/ 三态填充表 / dry-run 报告 / 历史面板一应俱全。

**但它从没真正点过火。** 翻 `phase-0-validation-worksheet.md` 与 shiny-wizard 计划执行记录,
两件决定生死的事仍是空白:① 草稿合格率从没在本站真题材上测过;② authorized 真实发布
一次都没跑通(Step 3「待用户执行」)。换句话说:这个"自主发布器"100% 造好、0% 验证。

操作者目标已确认:**自用,要能每天靠它替代手工发帖。** 最在意的风险:**草稿质量 / 幻觉**——
本站是 NSFW 里番介绍站,固定「51娘」口吻 + 逐作品事实 + 漢化/無修连结。AI 一旦编造
作品名、集数或连结,就是把错误信息发上自家域名,伤读者信任与 SEO,且难撤回。

因此本轮收敛方向 **不是更多功能**,而是:**(A) 源接地防幻觉改造 +(B) 一次真实首飞拿基线**。
已过度建设的可靠性 / 可观测性,本轮零新增。

## 新生成流程(源接地)

```
旧: 选题(一行) → AI 生成整帖(事实也由 AI 编) → 审核 → 发
                                  ↑ 幻觉风险在这里

新: 选题 + 事实块(你贴)
      作品名 / 集数 / 漢化·無修连结 / 其他事实
          ↓
      AI 只做:套 51娘口吻 + 组织成帖
          ↓  (规则:只用我给的事实;缺的标【待补】;严禁编造作品名/集数/连结)
      草稿(事实零幻觉,连结只来自输入)
          ↓
      审核区显著呈现【待补】标记 + 所有连结
          ↓
      人审 → authorized 发
```

## 分阶段结构(已定:先基线 · 闸门 · 再建)

本轮**不是**一口气做 A+B+C。决策已定为**前提先行**:

```
阶段 0(前提基线,~1–2h,几乎零写码)
  R0  重核后台契约
  RB  用「现版 prompt(无源接地)」跑 5–10 真题材
        → 标质量(直接发/小改/不合格,分开计)
        → 计时:事实从哪来、找多久、审改多久 vs 手工基线
        ↓
  ┌──────────── 闸门(看 RB 数据)────────────┐
  │ 若 幻觉确为瓶颈 且 找事实成本可控          │ → 阶段 1:建源接地(R4–R7)→ R8 迭代 → R2 真发
  │ 若 找事实才是瓶颈 / AI 只剩套壳没省时间    │ → 不建源接地,转向(如定向自动取链/模板化),重开 brainstorm
  └────────────────────────────────────────────┘
```

阶段 1 的具体单元(R4–R8)只有在闸门 = go 时才进入 `/ce:plan`。

## Requirements

**A. 阶段 0:前提基线 + 首飞验证(本轮必做,产出决策数据)**
- R0.(前置闸)在跑任何题材前**重新核对后台契约**:`POST /admin/webarticle/save` 提交机制、表单字段集、选择器、Quill/layui 行为是否与 2026-06-04 实证一致;有漂移先记录再修。这是唯一硬交付(R2)的外部依赖,核对成本极低,中途才发现成本极高。
- RB.(前提基线,**闸门依据**)用**现版 prompt(无源接地)**在 **5–10 个本站真实题材**上生成草稿,逐条标注**分开记**「直接发 / 需小改 / 不合格」,并对每条**计时**:事实搜集耗时(并记**事实从哪来**)、审改耗时;同一批记一个**手工撰写基线时间**对比。「需小改」以时间界定(如 < N 分钟)。→ 既证实幻觉是否真发生,又测准"找事实"是不是真瓶颈。
- R2.(首飞,阶段 0 即可做)走通**至少一条 authorized 真实发布**的完整端到端链路(生成 → 填充 → `POST /admin/webarticle/save` → 后台核对 → 轨迹记录),记录全程摩擦点(选择器漂移、Quill 真实行为、layui 提交等)。可用现版内容先打通管路。

**B. 阶段 1(仅闸门=go 才做):源接地生成(防幻觉核心)**
- R4. 输入模型从"纯选题"扩为"**选题 + 事实块**":每条题目可附结构化事实(作品名 / 集数·话数 / 漢化连结 / 無修连结 / 其他要点)。**注意这是贯穿消息契约的改动,非纯 UI**:`RUN_BATCH` 载荷、`BatchItem`、`ContentDraft`、`buildPrompt` 签名都要从"单 topic 字符串"扩为携带 facts;故"结构化表单 vs 约定分隔符"是契约决定,须 Resolve Before Planning(推荐分隔符约定起步,保留 textarea 心智、好扩到 5–10 条)。
- R5. Prompt 改造为「**只润色、不造事实**」契约:AI 仅使用操作者提供的事实 + 套 51娘口吻组织成帖;任何未提供的事实**留空并标【待补】**;严禁编造作品名 / 集数 / 连结。`buildPrompt` 须改签名接收 facts 与新 token,并定义**空 / 部分事实**如何渲入 prompt(逼模型输出【待补】而非自由发挥)。
- R5b.(事实状态边界)定义三态行为:**零事实** → 仍生成全【待补】草稿(还是输入处拦截/告警?);**部分事实** → 填已知、其余【待补】;并明确**全【待补】草稿是否计入 R1 合格率基线**(避免污染数字)。
- R6. **连结只能来自输入**:草稿中出现的任何 URL 必须能在操作者输入里找到来源,AI 自造的 URL 视为违规;审核区把【待补】标记与全部连结显著呈现,让人一眼看到缺口。**注意**:`sanitizeBody` 会原样放行远程 https 链接,故连结核验是**独立于消毒的另一道闸**——须定义从正文 HTML 抽 `<a href>`、与输入比对、未命中如何处置(拦/标/剥);匹配规则(精确 vs 归一化)load-bearing,须 Resolve Before Planning。
- R7. 51娘口吻 + 帖体结构以**少量 gold-standard 范例(few-shot)**固化,降低风格漂移。**存储决定**:因 R8 要单独调 few-shot,few-shot 应作为**独立 Settings 字段**(非塞进 `promptTemplate` 单串)。

**C. 阶段 1(仅闸门=go 才做):质量快速迭代(轻量)**
- R8. 草稿质量可快速迭代:操作者能改 system prompt / few-shot 范例并**重跑同一批题目**对比效果(单一模板,**非**逐条 override)。**对比 = 手工肉眼读两轮输出**;保存上轮草稿做并排 diff / 版本存储**本轮 OUT**(除非首飞证明肉眼比对是瓶颈)。**必须绕过重入闸**:`runBatch` 会用 `filterReentrantTopics` 过滤 `publishedTopics`/隔离题目——R2 真发后这些题目被永久拉黑,重跑会**静默丢弃**;R8 需走"只生成不发"的迭代通道,不查 `publishedTopics`。

## Success Criteria

- **跑通 ≥1 条 authorized 真实发布**:后台保存成功、扩展状态与后台一致、轨迹记录可校验(`verifyTrajectory()` ✓)。
- **合格率达标**:**「直接发」与「需小改」分开计**,达到操作者拍板的阈值(默认建议 直接发+小改 ≥80%,待确认);「需小改」须时间界定。
- **零编造**:抽查 N 条草稿,零编造连结、零编造作品事实——缺的事实一律是【待补】而非杜撰。
- **确实省时间(可测,非主观)**:在**真实(非预先备好)题材**上,源接地全流程**单帖净耗时(含事实搜集+粘贴+审改)≤ 手工撰写同题耗时**;否则视为未达标(详见 R1 计时)。

## Scope Boundaries(本轮明确不做)

- **不再加任何可靠性 / 可观测性功能**:批量状态机、轨迹、崩溃恢复、tombstone、历史面板已足够,本轮封存不动。
- **不做源数据自动抓取 / scrape**:事实由操作者手动贴入(YAGNI;自动抓取留待验证证明值得后再说)。
- **不做事后逐条勾验清单 UI**:已选"源接地·提前掉死"而非"事后兜住";仅做 R6 的轻量呈现,不建完整 checklist 校验流。
- **不做连结可达性网络探测**:MV3 / CORS 受限;仅做"格式合法 + 来源于输入"校验,不发 HEAD 请求验活。
- **不做多站点 / 对外分发 / 配置化 / 直接 API 发布器**:确认为单操作者自用,不投资多租户能力。

## Key Decisions

- **市场 = 自用每天能发**(非对外产品):不投资多租户 / 配置化 / 分发能力。
- **防幻觉走"源接地·提前掉死"**(非"事后兜住"):AI 只润色操作者给的事实,从源头消除编造空间;代价是每条帖多贴几行事实,操作者接受。
- **工程地基已过度建设**:本轮零新增可靠性 / 可观测性,集中火力在内容质量 + 首飞验证。
- **先建源接地、再首飞测**(⚠ 经评审待复核):原意是首飞只测源接地版本、不再证伪旧版幻觉。但评审指出两点风险——(a) 无旧版基线,则 R4(输入模型)、R5(prompt 契约)、R7(few-shot)**三杠杆同时变**,≥80% 若达成也无法归因到底哪个起作用;(b) 幻觉是"最被担心"的风险,未必是"最挡日常使用"的瓶颈。是否插一个轻量"前提基线"步骤,见 Resolve Before Planning 首条战略决定。

## Dependencies / Assumptions

- 假设操作者**愿意每条帖手动提供核心事实**,以换取事实零幻觉。
- 假设后台 `POST /admin/webarticle/save` 提交机制与字段集自 2026-06-04 实证以来未变。
- 假设单一改进版 system prompt + 少量 few-shot 足以稳住 51娘风格(R7/R8 验证;若不足,再议)。
